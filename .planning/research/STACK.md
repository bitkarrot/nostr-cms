# Stack Research — Email Newsletter for nostr-cms

**Date:** 2026-07-04

## Recommendation

Build the email layer as **TypeScript serverless functions on Vercel** (the project's existing deploy target — there's a Vercel deploy button in the README) calling **Resend** as the third-party email API, with **Supabase Postgres** as the subscriber database. Keep the existing Go/Swarm ("bkrelay") backend untouched; the email endpoints live under a new `/api/email/*` path served by Vercel functions, authenticated with the same NIP-98 pattern the scheduler already uses.

## Email API — Resend

**Use Resend** because:
- First-class TypeScript SDK + native React Email templates — matches the React/TS codebase and lets posts render as emails from the same JSX mental model.
- Clean API, minutes to integrate. 3,000 emails/mo free, $20/mo for 50,000 — fits a single-tenant creator newsletter.
- SOC 2 Type II, good deliverability on shared pool, 99.9% webhook uptime SLA.
- Per-email sending rate is controlled by a token-bucket limiter we own (Resend does not expose a "5–5000/min" admin knob), which is exactly what the requirement asks for.

**Rejected alternatives:**
- **SendGrid** — retired its free tier in May 2025; heavier SDK; no React Email.
- **Mailgun** — no ongoing free tier, manual SPF/DKIM/DMARC, community-maintained TS SDK, ~4x cost of Resend at entry.
- **Buttondown / Kit / Beehiiv** — these are hosted newsletter *platforms*, not send APIs. Using one means the list/audience lives in their UI, not nostr-cms, which breaks the requirement that admins manage subscribers (CSV upload, segments) inside nostr-cms.
- **AWS SES** — cheapest at volume but raw infra, no React Email, more setup; overkill for single-tenant creator scale.
- **Self-hosted MTA** — explicitly out of scope (deliverability + compliance burden).

## Server layer — Vercel Serverless Functions (Node runtime)

**Use Vercel Serverless Functions (Node.js runtime, not Edge)** because:
- Full npm ecosystem — Resend SDK, CSV parsers, zod, crypto all work.
- Up to 300s timeout — enough for a rate-limited batch send of a post to a large list.
- The project already deploys to Vercel; `/api/*` is the conventional function directory, matching the existing `/api/scheduler/*` and `/api/admin/*` proxy pattern.
- Colocated env-var secrets management (Resend API key, Supabase service role) never reach the client bundle.

**Why not Edge functions:** Edge runtime restricts to Web APIs and has a 10s timeout on Hobby — too short for rate-limited batch sends. The email send path needs full Node + long timeout.

**Why not extend the Go/Swarm backend:** Would require Go work and a Go Resend client, splitting the email codebase from the TS frontend. Reusing NIP-98 verification in a TS function keeps one language. The Go backend remains the source of truth for relay/scheduled-posts; email is additive.

## Subscriber DB — Supabase Postgres

**Use Supabase Postgres** because:
- Free tier, RLS for defense-in-depth, REST autogenerated (PostgREST), and the `supabase` skill is already available to the agent.
- SQL gives clean segment queries (Followers / Contributors / Reward buyers) and CSV bulk-insert.
- Edge Functions available later if we need a webhook receiver for Resend delivery events colocated with the DB.
- Schema can carry a `site_id` column from day one so multi-tenant is additive, not a rewrite (per the "single now, multi later" decision).

**Why not SQLite-in-Go-backend:** couples subscriber data to the relay host; harder to query from TS functions; no RLS.
**Why not IndexedDB-only:** subscriber data is server-side PII that must persist across users/devices and drive server-side sends — cannot live in the browser.

## Frontend (no new deps)

- Reuse shadcn/ui (Switch, Dialog, Form, Table, Slider) for admin + signup UI.
- Reuse react-hook-form + zod for signup/CSV forms.
- Reuse TanStack Query for subscriber list / send-status mutations.
- Add `resend` only as a server-side dep (Vercel function), never imported by the client bundle.

## Versions to pin (server-side, new)

- `resend` — latest stable published ≥ 7 days ago at install time.
- `@supabase/supabase-js` — latest stable.
- `csv-parse` (or `papaparse`) — for CSV ingest server-side.
- `zod` — already a client dep; reuse on server.

## Secrets layout (server-only env vars)

- `RESEND_API_KEY` — never in `VITE_*`.
- `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY` — service role, server only.
- `EMAIL_FROM_DOMAIN` — verified sending domain.
- Rate limit + module toggle stored in Supabase `settings` table, editable from admin UI (not env vars).
